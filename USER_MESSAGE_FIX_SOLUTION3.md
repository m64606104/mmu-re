# 用户消息保留方案3实现 + AI上下文bug修复

## 🎯 问题1：用户消息被刷新

### 问题描述
```
1. 用户点击"生成AI回复"
2. AI开始生成（需要5秒）
3. 用户在等待期间发送消息："等等，我还想问..."
4. ❌ 用户的新消息被删掉了
```

### 根本原因
```typescript
// 创建快照
let currentMessages = [...conversation.messages];

// 用户发新消息 → conversation.messages更新

// AI消息回调
onAIMessage(message) {
  currentMessages = [...currentMessages, message];  // 基于快照
  onUpdateConversation({
    messages: currentMessages  // ❌ 用快照覆盖，用户新消息丢失
  });
}
```

## ✅ 方案3：按时间顺序排列（已实现）

### 为什么选择方案3？

**用户的观点（✅ 正确）**：
> "我觉得方案三更贴近现实中群聊的对话。现实中有时候在群聊聊天的时候，用户会想要插入一些群聊中正在进行的讨论，就会出现类似于方案三的情况。这些插入的对话并不会第一时间就被回复，因为其他人还在进行原话题的讨论。"

**真实群聊场景**：
```

小红：爬山可以锻炼身体
小华：我也想去爬山
小刚的插话并不会立即被回应  ← 这就是真实场景
```

### 实现原理

**核心代码**（ChatScreen.tsx 第2747-2765行）：
```typescript
onAIMessage: (_aiId, message) => {
  // 🎯 方案3：从最新的conversation.messages获取，保持用户插话的时间顺序
  // 这样更符合真实群聊场景：用户可以在AI回复期间插话
  const latestMessages = conversation.messages;
  
  // 累积AI消息到快照（用于其他逻辑）
  currentMessages = [...currentMessages, message];
  
  // 合并：最新消息列表 + 新的AI消息
  // latestMessages可能包含用户在生成期间发送的消息
  const updatedMessages = [...latestMessages, message];
  
  console.log(`📨 AI消息追加: 原${latestMessages.length}条 → 现${updatedMessages.length}条`);
  
  onUpdateConversation(conversation.id, {
    messages: updatedMessages,
    lastMessageTime: Date.now()
  });
}
```

## 🐛 问题2：AI第二轮重复行为

### 问题描述

用户反馈：
> "目前第一轮之后的回复的时候ai对上下文的了解很奇怪，例如会出现一种情况：在第一轮的时候 小a起哄让小b快点发红包，然后小c也附和了起哄，于是小b发了红包。在进行下一轮的时候 小a还继续起哄让小b快点发红包"

**场景**：
```
第一轮：
小a：小b快发红包！
小c：是啊，快发！
小b：[发群红包]

第二轮（点击生成）：
小a：小b快发红包！  ← ❌ 红包已经发了，为什么还在催？
```

### 根本原因

**代码位置**（groupChatService.ts 第392行）：
```typescript
// 构建消息历史（最近20条）
const recentMessages = groupConversation.messages.slice(-20);
```

**问题分析**：
1. AI只看最近20条消息
2. 如果对话很长，红包消息可能在第21条之前
3. AI看不到"红包已发"这个事实
4. 所以继续催发红包

### 修复方案

**增加上下文窗口**（groupChatService.ts 第391-392行）：
```typescript
// 构建消息历史（最近30条，增加上下文窗口）
const recentMessages = groupConversation.messages.slice(-30);
```

**效果**：
- AI能看到更多历史消息（20条 → 30条）
- 更容易看到关键事件（如"红包已发"）
- 减少重复行为
